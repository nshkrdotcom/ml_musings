Before CS336, you are already strong on the **conceptual Transformer spine**. The remaining prep should be less “what is attention?” and more **can I implement, measure, train, debug, and reason about cost?**

Stanford CS336 is explicitly “Language Modeling from Scratch”; the current Spring 2026 schedule starts with tokenization, PyTorch/einops, FLOPs/memory/arithmetic intensity, architectures, attention alternatives, MoE, GPUs/TPUs, kernels/Triton/XLA, parallelism, scaling laws, inference, evaluation, data, and post-training/RLHF/RLVR. ([Stanford CS336][1]) The public course description says the course walks through the full LM creation process: data collection/cleaning, Transformer construction, training, evaluation, and deployment. ([Stanford Bulletin][2])

So I’d learn these next.

# 1. Python + PyTorch fluency

Your Elixir/Nx work maps conceptually, but CS336 assignments are Python/PyTorch-centered. Be comfortable with:

```text
torch.Tensor shapes
nn.Module
forward()
autograd
optimizers
DataLoader
torch.no_grad()
device placement
dtype: fp32, fp16, bf16
```

Also learn **einops**. CS336 explicitly puts PyTorch/einops in lecture 2. ([Stanford CS336][1])

Minimum target:

> Implement a tiny decoder-only Transformer in PyTorch without copying a tutorial line-by-line.

# 2. Tensor shape manipulation

You have Shape Literacy conceptually. Now make it mechanical.

Learn these cold:

```python
view / reshape
transpose
permute
contiguous
unsqueeze / squeeze
expand / repeat
matmul / bmm / einsum
torch.einsum
einops.rearrange
```

Especially for:

```text
{batch, seq, hidden}
→ {batch, heads, seq, head_dim}
→ attention scores {batch, heads, seq, seq}
→ output {batch, seq, hidden}
```

This is probably the highest ROI prep.

# 3. Cross-entropy language modeling

You need to internalize the exact next-token training objective.

The key shape pattern:

```text
logits:  {batch, seq, vocab_size}
targets: {batch, seq}
loss:    scalar
```

And the shift:

```text
input tokens:  [t0, t1, t2, t3]
target tokens: [t1, t2, t3, t4]
```

Learn why training can predict all next tokens in parallel with a causal mask, even though inference generates one token at a time.

# 4. Optimizers and training stability

You do not need optimizer-theory mastery, but you should know what each knob does when training blows up.

# 6. Resource accounting

This is a big CS336 theme: FLOPs, memory, and arithmetic intensity appear in lecture 2. ([Stanford CS336][1])

Learn to estimate:

```text

attention memory

KV cache size
tokens/sec
GPU memory bandwidth bottlenecks
```

A good prep exercise:

> Given a Transformer with layers L, hidden size D, sequence length T, vocab V, heads H, estimate parameter count and rough training memory.

# 7. GPU mental model

You do not need to be a CUDA wizard before starting, but you should know why a GPU is not “just a faster CPU.”

# 12. Scaling laws

CS336 spends multiple lectures on scaling laws. ([Stanford CS336][1])

Prep the ideas:

```text

```

You do not need to derive everything first, but know what question scaling laws answer:

> Given a compute budget, how big should the model be and how many tokens should it train on?

# 14. Transformer variants

You already know the vanilla block. Add:

```text

```

CS336 includes architectures/hyperparameters, attention alternatives, and MoE early in the schedule. ([Stanford CS336][1])

# 15. Post-training basics

CS336 includes SFT/RLHF/RLVR/alignment later in the course. ([Stanford CS336][1])

Prep only the map:

```text
pretraining = next-token prediction on broad corpus
SFT = supervised examples of desired behavior
preference modeling = learn which response is better
RLHF/RLVR = optimize policy using reward/verifier signal
DPO/GRPO/PPO = common post-training families
```

No need to go deep before CS336, but know the phases.

# Your best pre-CS336 sequence

Given what you have already done, I would not add more abstract theory first. I would do this:

```text
1. PyTorch crash course
2. einops + shape drills
3. implement BPE tokenizer
4. implement tiny GPT from scratch
5. train it on a tiny corpus
6. add sampling + KV cache
7. measure parameter count, FLOPs, memory
8. read CS336 lecture 1–2 and start assignments
```

The key implementation project:

> Build a tiny decoder-only Transformer that can overfit a tiny text file.

That one project forces together:

```text
tokenization
embeddings
positions/RoPE
causal attention
MLP
residuals
cross-entropy
AdamW
sampling
loss curves
GPU memory
```

# What you do **not** need first

You do not need to master all of these before starting CS336:

```text
measure theory
advanced convex optimization
full CUDA programming
RLHF math
distributed systems at production scale
Triton kernel writing
```

CS336 will expose you to kernels, parallelism, scaling, and post-training, but you can learn those while doing the course.

# Bottom line

You are conceptually ready for the Transformer parts. The missing readiness is mostly:

```text
PyTorch implementation fluency
resource accounting
tokenizer/data pipeline practice
training loop debugging
GPU/memory intuition
```

Once you can implement and train a tiny GPT in PyTorch, CS336 becomes challenging but tractable instead of overwhelming.
